One Bed Remote Monitoring Build Plan

A focused build spec for the actual product: one private room, one monitored bed, live remote video, event detection, evidence capture, VLM interpretation, and a grounded chat assistant.

Scope: one private room Primary surface: remote monitoring console Mode: clinician-in-the-loop Version: May 2026 reset

1. What We Are Actually Building

This product is a remote patient monitoring system for one private hospital room. The remote user must be able to see the live room feed, understand what is happening, review detected events, inspect evidence, and ask a grounded assistant questions about the patient and the room.

The product loop: live video feed -> detection -> event timeline -> evidence clip -> VLM review -> grounded assistant -> staff alert or chart note.

This means the live video feed is not a nice-to-have. It is the base layer. Detection, VLM, alerts, and assistant all sit on top of the remote monitoring feed.

V1 Now Means One Bed, Not A Hospital Platform

  • One private room.
  • One bed-zone polygon.
  • One primary room camera feed, usually RTSP/IP camera or a local webcam during development.
  • Optional second camera later for monitor close-up or round/tablet capture.
  • One remote monitoring dashboard.
  • One patient timeline.
  • One assistant grounded in video events, OCR vitals, patient context, and evidence.

5. Video Feed: The First Real Requirement

The first build must prove that a remote user can see the private room video feed from another machine/browser.

Video Feed Requirements

  • Connect to a camera source: webcam, RTSP camera, phone camera RTSP app, or video-file simulator.
  • Display the live feed in the remote monitoring console.
  • Show connection state: live, reconnecting, offline, low FPS.
  • Allow snapshots for evidence capture.
  • Maintain a rolling clip buffer, for example 30 seconds before and 30 seconds after an event.
  • Support privacy masking later, but do not block the first feed milestone on it.

WebRTC vs HLS

Option Use It When Tradeoff
WebRTC You need low latency and possible two-way communication. More setup, but best remote monitoring feel.
HLS You want simpler browser playback. Higher latency, usually several seconds.
MJPEG You want a quick development proof. Simple, but inefficient and not final.
Build order correction: do not start with the chatbot. Start with camera feed visible remotely, then event timeline, then detection, then VLM, then chat assistant.

6. Detection Scope For The One-Bed Build

The detection system should produce candidate events. Some events can be deterministic, some need VLM confirmation, and some require human review.

Event Signal Trigger Output Review
bed state Person position vs bed polygon. In bed, sitting edge, out of bed, on floor. Timeline event and state panel update. Review only if critical or uncertain.
on floor / fall Pose, bbox height, floor zone, movement pattern. Patient centroid below bed/floor line or horizontal posture near floor. Critical alert, evidence clip, VLM confirmation. Always review.
no movement Motion magnitude in patient region. No meaningful movement for configured threshold. Warning event and timeline marker. Review if prolonged or paired with abnormal vitals.
distress / breathing problem Chest motion, posture, agitation, face/upper body cues, monitor vitals. Elevated work of breathing, repeated abnormal RR/SpO2, visible distress. Warning or critical event with explanation. Always review before clinical wording.
staff presence Second person in room, uniform cues, region occupancy. Staff enters/exits or no staff visit for threshold. Care-gap log and visit history. Review for compliance reports.
monitor OCR Monitor crop OCR. Scheduled every 30-60 seconds or when monitor visible. HR, BP, SpO2, RR readings with confidence. Review low-confidence or abnormal values.
oxygen interruption Visible cannula/mask/tubing, SpO2 trend, staff intervention. Mask/cannula displaced, oxygen tube disconnected, SpO2 drop. Warning event with evidence. Always review initially.
IV status IV bag crop, pump screen, line visibility. Near empty, stopped, disconnected, occluded. Info/warning event. Review until model is reliable.

7. VLM Layer

The VLM layer should not run on every frame. It should be called when the deterministic system needs help, or at a low scheduled interval for scene summaries.

When To Call The VLM

  • Candidate fall/on-floor event.
  • Possible respiratory distress or visible breathing problem.
  • Oxygen tubing/mask/cannula appears displaced.
  • IV state is unclear.
  • Staff/patient role is ambiguous.
  • Every 60-120 seconds for a low-cost room summary if budget allows.

VLM Output Contract

{
  "scene_summary": "Patient is lying in bed, oxygen cannula visible, no staff present.",
  "patient_state": "in_bed",
  "distress": {
    "suspected": false,
    "reason": "No visible labored breathing; SpO2 OCR is 96 if accepted."
  },
  "fall_or_floor": {
    "suspected": false,
    "confidence": 0.08
  },
  "oxygen": {
    "visible": true,
    "issue_suspected": false,
    "reason": "Cannula appears positioned on face."
  },
  "iv": {
    "visible": true,
    "state": "unclear",
    "reason": "Bag is partly occluded."
  },
  "staff_present": false,
  "recommended_event": null,
  "needs_human_review": false
}
Key rule: VLM output becomes evidence and explanation. It does not directly become clinical truth until reviewed or corroborated by deterministic signals.

8. Chat Assistant Layer

The assistant should answer questions using the video-derived event timeline, vitals OCR, evidence clips, patient context, and notes. It should not guess from memory.

Assistant Tools

Tool Purpose
get_current_room_state Returns live bed state, staff presence, camera health, active alerts.
get_event_timeline Returns events for selected time window.
get_vitals_trend Returns accepted OCR vitals and abnormal streaks.
get_event_evidence Returns still frame, clip URL, rule trace, VLM output, review status.
create_review_note_draft Creates a draft note for human sign-off.
send_alert Routes a staff alert only after user confirmation or configured policy.

Questions It Must Handle

  • "What is happening in the room right now?"
  • "Has the patient moved in the last 3 hours?"
  • "Show me the evidence for the fall alert."
  • "Was staff present after the oxygen issue?"
  • "Summarize overnight events."
  • "Did SpO2 drop before the distress alert?"
  • "What should I review first?"

Assistant Guardrails

  • Always cite event IDs, vitals timestamps, or evidence clips for factual claims.
  • Say "not visible" or "not enough evidence" when the camera view is insufficient.
  • Never say an alert was sent unless the alert system confirms delivery.
  • Never create a final clinical order. Create a draft for clinician review.
  • Separate observed facts from model interpretation.

9. Evidence And Review Workflow

Every important detection should become a reviewable event, not just a notification.

Event Lifecycle

  1. Frame or motion pattern creates a candidate event.
  2. State machine checks thresholds and history.
  3. Evidence service saves still frame and rolling clip.
  4. VLM confirms or explains if needed.
  5. Event appears on remote console timeline.
  6. Alert routes if severity requires it.
  7. Human reviews: true positive, false positive, unclear, resolved.
  8. Assistant can summarize the event with citations.
  9. Reviewed event can become a chart note draft.

Review Actions

Action Meaning
Acknowledge User saw the event and takes responsibility for next step.
Mark true positive Event was correct.
Mark false positive Event was incorrect and should be used for evaluation.
Escalate Notify assigned nurse, station, or remote clinician.
Create note draft Draft a timeline note, never final chart write without sign-off.
Mute rule temporarily Pause a noisy rule for a defined time and reason.

12. Build Sequence For Right Now

This is not a generic product roadmap. This is the immediate order for the one-bed build.

  1. Remote video proof: one camera source appears in the browser from another device or browser session.
  2. Room state shell: console shows camera live/offline, bed state placeholder, and timeline placeholder.
  3. Evidence capture: user can click "capture evidence"; system stores a still and short clip.
  4. Bed-zone calibration: draw bed polygon on a frame and save it.
  5. Basic detection: in bed, out of bed, on floor using frame analysis or prototype state machine.
  6. Event timeline: detections appear as events with status and evidence.
  7. Monitor OCR: crop monitor area, extract HR/BP/SpO2/RR, show accepted readings.
  8. Distress candidates: combine visible posture, motion, RR/SpO2 OCR, and VLM scene check.
  9. IV/oxygen candidates: detect visible issues; mark low-confidence until reviewed.
  10. VLM confirmation: call VLM for fall, distress, oxygen, IV, and ambiguous events.
  11. Review workflow: acknowledge, true positive, false positive, escalate, note draft.
  12. Ask AINA: chat can answer questions using current state, timeline, vitals, and evidence.

13. What The Earlier Comprehensive PRD Still Gives Us

The broader PRD was useful, but it should be treated as future expansion. The one-bed build should reuse these parts now:

  • AINA-style command center interaction pattern.
  • Event timeline with evidence and charting review.
  • Ask AINA panel grounded in patient context.
  • Vitals side panel and trend view.
  • Bedside call and alert routing later in the same console.
  • Morning digest after event history exists.

Do not start with multi-bed grids, hospital-wide admin, complex FHIR, or mobile apps. Those become useful only after one bed works end to end.

14. Definition Of Done For The First Real Demo

Demo goal: someone opens the remote console and sees one live private room feed. The system detects a patient leaving the bed, records the event, saves evidence, runs VLM review, and the assistant can explain what happened using that evidence.

Checklist

  • Live video visible remotely.
  • Camera status displayed.
  • Bed zone calibrated.
  • At least three bed states detected: in bed, out of bed, on floor.
  • Event timeline updates live.
  • Evidence still and clip saved for event.
  • VLM structured output attached to event.
  • Monitor OCR can read sample vitals.
  • Assistant answers "what happened?" with event ID and evidence reference.
  • Human can mark event true positive or false positive.

15. Things To Be Careful About

Do Not Overbuild

One bed first. No hospital platform until the one-bed loop works.

Do Not Hide Uncertainty

Every detection needs confidence, rule trace, and review state.

Do Not Make Diagnosis Claims

Say "possible distress detected" rather than "patient has respiratory failure."

Do Not Upload Everything

Upload event evidence, not continuous raw video, unless explicitly configured.

16. Final Product Shape

The correct product is a live remote monitoring console with intelligence layered onto it:

  1. See the room live.
  2. Know the current state.
  3. Detect important changes.
  4. Record evidence.
  5. Use VLM to interpret ambiguous scenes.
  6. Use chat to query what happened and what needs review.
  7. Route alerts and draft notes only after the system has evidence.

That is the build. Everything else is expansion.
